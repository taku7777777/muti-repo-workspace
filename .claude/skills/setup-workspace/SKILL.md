# setup-workspace

Initializes (or refreshes) this workspace. Everything is driven by
`config/repos.json` and `config/workspace.json`.

## Steps

1. **Pre-flight.** Read `config/repos.json` and `config/workspace.json` and
   sanity-check them:
   - Any repository `url` still containing `your-org` is placeholder content —
     tell the user to edit `config/repos.json` first (the script will skip
     those entries, so setup can still proceed for real entries).
   - If `allowed_push_orgs` is empty, mention that pushes will be
     unrestricted until they add their org(s).

2. **Run the script** from the workspace root:

   ```bash
   bash scripts/setup-workspace.sh
   ```

   It is idempotent: existing clones are kept (only `extensions.worktreeConfig`
   is re-asserted), settings/hook/helpers are re-generated.

   If cloning fails due to sandbox network restrictions (SSH to github.com may
   be blocked inside the sandbox), ask the user to run the same command in a
   normal terminal — suggest typing `! bash scripts/setup-workspace.sh` in the
   prompt.

3. **Report** what was done: repos cloned/skipped, settings written, hook
   installed, helpers installed. Then remind the user:
   - Restart `claude` in the workspace root so `.claude/settings.json`
     (generated by the script) takes effect, and **accept the trust dialog** —
     if it is not trusted, Claude Code ignores `permissions.allow` entries and
     the sandbox setup is incomplete.
   - Next command: `/open-task`.

## Notes

- Never edit files under `repositories/` — they are worktree origins.
- The root settings template is `templates/root/claude-settings.json`; change
  the template (via PR), not the generated file.
